Design Notes
Amzi! Prolog as part of the Eclipse Modelling Framework (EMF)

EMF is about transformations. A model can be expressed either as Java interfaces, UML, or XML. EMF translates between them.

Prolog, with it's powerful pattern-matching engine, is an ideal tool for transformations.

In particular, making a transform from the XML output of EMF to working Prolog code would not be difficult.

Proposal for one prototype:

Write a Prolog translator that takes the XML output from EMF for the EMF tutorial examples, and translates it into working Prolog code that can then be executed in the Prolog perspective of Eclipse running alongside the EMF perspectives.

Because the fundamental designs from EMF are object-oriented (OO), this would require using OO techniques in the generated Prolog code. The existing Amzi! frame implementations will be used as a first step in this modelling.

Proposal for another:

Write a Prolog translator that takes the tutorial SBVR example and translates the SBVR specification into working Prolog code.

Thoughts on ATL use case, natural language syntax to SBVR transformation:

http://www.eclipse.org/m2m/atl/usecases/Syntax2SBVR2UML/

In this case, the Atlas Transformation Language (ATL) http://wiki.eclipse.org/ATL/User_Guide is used to translate between a UML syntactic model of a sentence and a UML model of Semantic Business and Vocabulary Rules (SBVR).

It is interesting that there are a number of tools being developed for the sole purpose of doing model transformations. The main advantages touted for ATL are that it is a mixture of "declarative and imperative programming." This, of course, is exactly what Prolog provides as well. I cannot comment on the relative strengths and weaknesses of the various tools, but can assert that Prolog is very good at this sort of thing.

The parsing of a sentence into it's syntactic units, noun phrases, verb phases, etc., is exactly what Prolog was initially designed for.

Thoughts on ATL use case, transforming multiple rule langugase:

http://www.eclipse.org/m2m/atl/usecases/SharingRulesBetweenOCLUML_SWRLOWL/

OMG (not Object Management Group) what a lot of rule languages and models there are! And transformation languages!

It's clear there are many people working in the field of tools for expressing knowledge, and mapping between the various tools.

It appears Prolog is often used as the development tool underneath these various tools and representations. Using Prolog directly as the tool for our own implementations makes sense because Prolog is extensible. That is, if one needs OO capabilities, then they can be added in Prolog, or if one needs production rule reasoning, then that can be easily added in Prolog as well.

The use of Prolog directly is the difference between using an application and using a programming language. With an application, you are constrained to what the application provides. But with a programming language, you can make it do anything you want, with more or less difficulty depending on the language.

Without fully understanding the various tools, such as R2ML, SWRL, etc., I would guess that Prolog provides a lower level language that therefor gives us more flexibility for the modelling tasks we envision.
